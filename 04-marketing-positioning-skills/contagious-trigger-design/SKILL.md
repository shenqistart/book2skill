# 疯传触发设计

## R — 原文 (Reading)

> “社交货币。”
>
> “诱因。”
>
> “故事。”
>
> 来源：《疯传》STEPPS 框架。

## I — 方法论骨架 (Interpretation)

疯传不是随机好运，而是内容拥有被分享的理由。

用户分享一件事，通常不是为了帮品牌，而是为了表达自己、回应触发、传递情绪、展示公共行为、提供实用价值，或讲一个好故事。
STEPPS 提供六个检查点：社交货币、诱因、情绪、公共性、实用价值、故事。
不是每个内容都要六项全满，但至少要有清楚的传播发动机。
如果内容只对品牌重要，对用户没有谈资、场景和复述结构，就很难自然扩散。

这个 skill 的核心是把“做爆款”拆成可诊断的传播触发。

## A1 — 书中的应用 (Past Application)

### 案例 1：社交货币

- **问题**：人们为什么愿意谈论某些品牌或内容。
- **方法论的使用**：判断分享是否能让用户显得有趣、聪明、有品位或掌握内部信息。
- **结论**：传播要给分享者身份收益。
- **结果**：内容从品牌自说自话变成用户愿意拿来表达自己。

### 案例 2：诱因和故事

- **问题**：有些内容很有趣，但很快被忘记。
- **方法论的使用**：设计高频触发场景，并把产品嵌入故事。
- **结论**：传播需要被生活场景反复唤起。
- **结果**：内容不只是看过，而是被触发后再次讲述。

## A2 — 触发场景 (Future Trigger)

### 用户会在什么情境下需要这个 skill?

1. 用户想提高文章、短视频、活动、产品发布或品牌事件的自然传播。
2. 用户问“为什么没人转发/没人讨论”。
3. 用户想把产品功能设计成可分享。
4. 用户需要检查内容是否有社交货币、情绪和故事。

### 语言信号

- “怎么让这个内容更容易传播？”
- “为什么用户不愿意转发？”
- “这个活动有疯传潜力吗？”
- “怎样设计分享点？”

### 与相邻 skill 的区分

- 与 `diffusion-tipping-diagnosis` 的区别：本 skill 看内容传播要素；后者看网络关键人、附着力和环境临界。
- 与 `participation-word-of-mouth-loop` 的区别：本 skill 处理传播载体，后者处理长期用户参与机制。

## E — 可执行步骤 (Execution)

1. **定义分享对象**
   - 完成标准：明确要被分享的是内容、活动、产品功能、观点还是品牌事件。

2. **逐项检查 STEPPS**
   - 完成标准：为社交货币、诱因、情绪、公共性、实用价值、故事各打高/中/低。

3. **找出主发动机**
   - 完成标准：选 1-2 个最有潜力的传播发动机，不硬凑六项。

4. **改写分享句**
   - 完成标准：用户能用一句自己的话转述它为什么值得分享。

5. **检查品牌嵌入**
   - 完成标准：传播时品牌或产品不会被故事剥离。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 用户想购买流量、刷屏或操纵榜单。
- 内容传播会放大虚假、恐慌、歧视或隐私风险。
- 产品价值不足，只想用噱头制造关注。

### 作者在书中警告的失败模式

- 只做有趣内容，品牌被用户忘记。
- 情绪很强但缺少产品嵌入。
- 分享理由只服务品牌，不服务分享者。

### 作者的盲点 / 时代局限

- 平台算法、审核和推荐机制会改变传播路径。
- 不同文化语境中“社交货币”和情绪表达差异很大。
