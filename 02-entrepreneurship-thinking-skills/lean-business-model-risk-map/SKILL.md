# 精益商业模式风险地图

## R — 原文 (Reading)

> “最好的方法是先把商业模式写在一页纸上。”
>
> “先验证计划中风险最大的部分。”
>
> “资金及其流动性会影响初创企业成败。”
>
> 来源：《精益创业实战》《创业之殇》。

## I — 方法论骨架 (Interpretation)

这个 skill 把“商业模式怎么写”改写成“哪一块最可能杀死项目”。

第一，商业模式不是 BP 叙事，而是客户、问题、价值主张、解决方案、渠道、收入、成本、关键指标和优势之间的闭环。
第二，早期不是平均推进所有模块，而是找到最大未知和最大风险。
第三，风险通常不只在产品：市场是否真实、客户是否可达、收入是否覆盖成本、执行是否跟得上、现金是否撑得住。
第四，失败案例显示，漂亮增长可能掩盖单位经济、资本依赖、组织能力或现金流断裂。

它适合在 MVP 之后或融资前，帮助 agent 把创业项目从故事变成风险排序表。

## A1 — 书中的应用 (Past Application)

### 案例 1：《精益创业实战》的精益画布

- **问题**：传统商业计划书太长，难以及时暴露风险。
- **方法论的使用**：用一页图呈现商业模式组件，并按风险排序。
- **结论**：先写 A 计划不是为了证明自己对，而是为了知道先测试什么。
- **结果**：计划变成实验清单。

### 案例 2：《创业之殇》的四类失败问题

- **问题**：初创公司失败常被归因于产品不好，但真实原因更复杂。
- **方法论的使用**：把失败拆到市场、产品、执行、资金和流动性。
- **结论**：商业模式风险需要跨模块检查。
- **结果**：避免只修产品、不看渠道和现金。

### 案例 3：《十亿美金的教训》的模式反例

- **问题**：明星项目增长很快，为什么仍会崩塌。
- **方法论的使用**：检查客户价值、成本结构、资本依赖和团队分裂。
- **结论**：融资和曝光可能延迟问题暴露，但不会消除模式缺陷。
- **结果**：风险地图成为失败复盘的前置工具。

## A2 — 触发场景 (Future Trigger)

### 用户会在什么情境下需要这个 skill?

1. 用户要写或审视一个创业商业模式。
2. 用户已经验证了部分需求，但不知道下一步验证哪块。
3. 用户准备融资，却缺少风险证据和单位经济假设。
4. 用户的项目增长不错，但现金、留存、获客成本或组织执行出现异常。

### 语言信号

- “商业模式哪里最危险？”
- “融资前应该补哪些证据？”
- “精益画布怎么填？”
- “这个模式看起来增长快，但会不会不可持续？”

### 与相邻 skill 的区分

- 与 `validated-learning-loop` 的区别：后者做单轮实验；本 skill 管理整张商业模式风险图。

## E — 可执行步骤 (Execution)

当 skill 被激活后，agent 应按以下步骤执行：

1. **画出一页模型**
   - 完成标准：列出客户、问题、价值主张、解决方案、渠道、收入、成本、指标、优势。

2. **标注证据等级**
   - 完成标准：每一格标为已验证、部分验证、假设、未知。

3. **排序最大风险**
   - 完成标准：按“为假时的致命程度 × 当前证据薄弱度”选出前三风险。

4. **设计验证顺序**
   - 完成标准：为每个高风险项绑定一个实验、指标和判停条件。

5. **输出经营建议**
   - 完成标准：给出先补证据、继续投入、收缩范围、暂停融资叙事或进入危机处理的建议。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 用户只要一份好看的 BP 文案。
- 用户要求实时融资条款、估值、法律或税务建议。
- 用户已经处在现金断裂、裁员或重大危机中，应先用 `hard-things-wartime-leadership`。

### 作者在书中警告的失败模式

- 用计划书替代证据。
- 先验证最容易的功能，而不是最大风险。
- 用融资成功掩盖商业模式未闭环。

### 作者的盲点 / 时代局限

- 一页画布会简化复杂供应链、监管和多边平台问题。
- 有些长期技术项目早期无法完整验证收入模型。
- 风险排序需要持续更新，不能一次填写后冻结。

### 容易混淆的邻近方法论

- “商业计划书”：本 skill 不是写作模板，而是风险地图。
- “财务模型”：本 skill 只做早期假设和风险排序，不替代详细财务建模。
