# 产品气质与表达

## R — 来源

依据《气质篇》《UI篇》中关于产品气质、人文意识、欢迎页、文字表达、条理清晰、少用 Tips、每个界面一个主题等论述。

## I — 方法论骨架

产品会通过每个字、每个默认项、每个提示、每个入口和每个视觉选择说话。
有气质不是装文艺，而是产品价值观自然流露。
UI 的第一原则不是绚丽，而是条理清晰；文案的第一原则不是讨好，而是像朋友一样准确沟通。
如果功能需要大量文字解释，往往说明设计还没有做到位。
产品应像作品，有统一人格、克制表达和清晰审美，而不是各部门妥协后的拼贴物。

## A1 — 书中的应用

### 案例: 欢迎页不罗列功能

- **问题**: 新版本欢迎页是否列出 1、2、3、4、5 个功能。
- **方法论的使用**: 作者把欢迎页视为少数能表达产品感受的地方。
- **结论**: 好产品不靠功能清单证明自己。
- **结果**: 微信欢迎页偏向场景、情绪和产品理念表达。

### 案例: UI 条理清晰

- **问题**: UI 是否应该追求更绚丽。
- **方法论的使用**: 作者强调主题、默认按钮、一致控件和最常用操作。
- **结论**: 视觉表现不如条理清晰重要。
- **结果**: UI 服务于体验，而不是为了展示设计能力。

## A2 — 触发场景

1. 用户要审查产品文案是否像人说话。
2. 用户要设计欢迎页、空状态、新功能提示或功能说明。
3. 用户要判断 UI 是否过度解释、过度炫技或没有主题。
4. 用户想让产品表达更有气质但不想变成营销腔。

### 语言信号

- “产品气质”
- “欢迎页怎么写”
- “这个文案像不像人话”
- “要不要弹 Tips”
- “UI 太花/太乱”
- “怎么做得像作品”

### 与相邻 skill 的区分

- 与 `scenario-boundary-design` 的区别: 后者判断功能是否打扰，本 skill 判断表达是否体现价值观。
- 与 `product-structure-evolution` 的区别: 后者处理骨架，本 skill 处理产品如何说话和呈现。

## E — 可执行步骤

1. **识别产品想表达的价值观**
   - 完成标准: 用 1-3 个词说明产品相信什么、拒绝什么。

2. **检查表达一致性**
   - 完成标准: 审查文案、视觉、提示、默认按钮、入口和空状态是否像同一个产品说话。

3. **减少解释和打扰**
   - 完成标准: 标出可删除的 Tips、功能罗列、成功废话、暧昧语气和强迫句式。

4. **强化主题和条理**
   - 完成标准: 每个界面或文案只保留一个主动作、一个主题和清晰完成路径。

5. **输出改写建议**
   - 完成标准: 给出保留原则、删除项和 1-3 个改写版本。

## B — 边界

- 不做脱离产品体验的广告口号。
- 不用“气质”掩盖可用性问题。
- 安全、合规、支付、医疗等场景可能必须显式说明。
- 审美判断不能只依赖个人偏好，应结合目标用户和使用场景。

### 失败模式

- 欢迎页罗列功能指标。
- 到处弹 Tips 教育用户。
- 文案讨好、暧昧、强迫或机器腔。
- UI 炫技但主题不清。
- 各模块像不同团队拼出来的。
