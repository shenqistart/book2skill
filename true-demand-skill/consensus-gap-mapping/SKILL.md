# 共识缺口映射

## R — 来源依据 (Reading)

> 原书依据见本节来源说明。

依据“特性与属性”“用户人设”“产品场景匹配”“利益驱动”“认知战”“关系”等章节。

本仓库不收录原书全文；此处只保留章节级依据和方法论重述。

## I — 方法论骨架 (Interpretation)

- 产品被购买和使用，前提是用户与市场形成某种共识。
- 共识失败常发生在六个层面：特性被看见但属性没被理解，用户人设错位，场景不匹配，利益不清，主张没有穿透认知，关系无法沉淀。
- 共识不是说服所有人，而是让目标用户在关键场景里知道这个产品“对我意味着什么”。
- 做产品时要把分歧显性化，而不是用更多功能掩盖分歧。

## A1 — 书中的应用 (Past Application)

### 案例 1: 手机品牌案例
- **问题**: 同样是智能手机，不同品牌被用户感知成完全不同的属性。
- **方法论的使用**: 从特性、属性和用户人设拆分感知差异。
- **结论**: 产品竞争不是参数竞争，而是共识竞争。
- **结果**: 魅族、小米、OPPO、苹果等案例展示了不同共识路径。

### 案例 2: 认知战案例
- **问题**: 新主张要进入用户认知，不只是发布一个口号。
- **方法论的使用**: 用利益、场景、关系反复支撑新主张。
- **结论**: 共识必须能被用户复述和使用。
- **结果**: 品牌关系成为共识结果。

## E — 可执行步骤 (Execution)

1. **列出分歧层**
   - 完成标准: 分别写下特性、属性、人设、场景、利益、关系六层当前说法。

2. **定位最大断点**
   - 完成标准: 找出哪一层最影响购买或使用行为。

3. **重写共识句**
   - 完成标准: 把产品主张改写成目标用户能在具体场景中复述的一句话。

4. **设计验证动作**
   - 完成标准: 确定用访谈、转述测试、落地页或小规模售卖验证共识。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 不要把共识等同于品牌口号。
- 如果产品价值本身不存在，共识工作会变成包装。
- 内部共识不能替代外部市场验证。

### 作者的盲点 / 时代局限

- 书中案例大量来自消费、手机、新消费和平台语境，B2B、深科技、监管产品需要另加约束。
- 价值和共识框架解释力强，但在早期极小样本阶段容易被过度概念化。
