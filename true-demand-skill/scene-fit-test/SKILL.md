# 场景匹配测试

## R — 来源依据 (Reading)

依据“用户人设”“场景的分歧：产品场景匹配”等章节。

本仓库不收录原书全文；此处只保留章节级依据和方法论重述。

## I — 方法论骨架 (Interpretation)

- 同一个功能在不同用户人设和场景下会变成不同需求。
- 产品场景不是“什么时候打开 App”这么浅，而是用户处境、关系、利益、时间、心理状态的组合。
- 场景错配会让产品看起来有用却卖不动、有人夸却没人买。
- 测试场景匹配时，要同时检查用户是谁、处境是什么、当下任务是什么、为什么现在需要。

## A1 — 书中的应用 (Past Application)

### 案例 1: SHEIN 与快时尚问题
- **问题**: 为什么快时尚和奢侈品能出现超级企业，中档产品却难。
- **方法论的使用**: 把不同消费场景、供应链速度和用户预期放在一起看。
- **结论**: 场景决定价值组合是否成立。
- **结果**: 快时尚场景支持高频、低价、快速更新。

### 案例 2: 新能源车降价问题
- **问题**: 为什么新能源车企会面临降价压力。
- **方法论的使用**: 从产品场景、基础设施、用户预期和商业模型推演。
- **结论**: 场景变化会重写最优商业模型。
- **结果**: 降价不是单一价格战，而是模式变化。

## A2 — 触发场景 (Future Trigger)

### 用户会在什么情境下需要这个 skill?

当用户想判断一个产品是否真的贴合目标人群和使用场景，或发现需求在不同场景里表现完全不同时调用。不适用于纯视觉审美判断。

### 语言信号

- "用户画像看起来对，但为什么没人用？"

## E — 可执行步骤 (Execution)

1. **写出场景四元组**
   - 完成标准: 明确用户、处境、任务、触发时刻。

2. **检查场景压力**
   - 完成标准: 判断时间、成本、情绪、风险中哪个压力最大。

3. **匹配价值类型**
   - 完成标准: 把场景压力与功能、情绪、资产、品牌价值对应。

4. **判定是否错配**
   - 完成标准: 输出“匹配/弱匹配/错配”及下一步验证方式。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 不要用虚构人设代替真实场景。
- 低频高风险场景需要更高证据标准。
- 场景匹配不能单独证明商业可行。

### 作者的盲点 / 时代局限

- 书中案例大量来自消费、手机、新消费和平台语境，B2B、深科技、监管产品需要另加约束。
- 价值和共识框架解释力强，但在早期极小样本阶段容易被过度概念化。
