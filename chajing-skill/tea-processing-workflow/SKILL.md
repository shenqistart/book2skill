# 制茶流程检查

## R — 来源

源文件: `chajing-skill/source/chajing.md`。

依据《三之造》“晴，采之，蒸之，捣之，拍之，焙之，穿之，封之”，并参考《二之具》的工具系统和《九之略》的可省条件。

## I — 方法论骨架

陆羽把制茶写成一条连续工序，而不是零散技巧。
采摘要看月份、天气、芽叶状态；加工要按蒸、捣、拍、焙、穿、封推进。
每一步都影响后一步，前面出错，后面不能靠包装补救。
工具也不是孤立名物，而是服务于流程的节点。
如果处于山园、野寺等简化场景，可以省部分工具，但不能省掉“成茶干燥、保存、避免败味”的核心目标。

## A1 — 书中的应用

### 案例: 从采摘到封藏
- **问题**: 茶如何从新叶变成可保存和饮用的茶。
- **方法论的使用**: 作者给出采、蒸、捣、拍、焙、穿、封的顺序。
- **结论**: 成茶质量来自连续流程控制。
- **结果**: 可转成制茶审查清单。

## A2 — 触发场景

1. 用户想复原《茶经》制茶流程。
2. 用户要检查某个制茶描述缺了哪一步。
3. 用户要做茶文化课程或演示，需要流程脚本。

### 语言信号

- “采蒸捣拍焙穿封”
- “按《茶经》怎么制茶”
- “这个制茶流程缺什么”
- “哪些工具是制茶用的”

### 与相邻 skill 的区分

- 与 `tea-quality-diagnosis` 的区别: 本 skill 检查过程；品质诊断还要看产地、形态和风险。
- 与 `contextual-simplification` 的区别: 本 skill 给完整流程；简化取舍另行判断。

## E — 可执行步骤

1. **确认场景**
   - 完成标准: 明确是完整复原、教学演示、流程审查还是简化实践。

2. **列出七步流程**
   - 完成标准: 按采、蒸、捣、拍、焙、穿、封列出输入、动作、输出。

3. **检查缺口和风险**
   - 完成标准: 标注天气、采摘、火候、干燥、保存的风险点。

4. **判断是否可简化**
   - 完成标准: 若用户处于野外/教学/演示场景，提示调用 `contextual-simplification`。

## B — 边界

- 不输出现代工业加工参数。
- 不把唐代饼茶流程当作所有现代茶类通用流程。
- 不在食品生产合规场景中替代专业标准。

### 失败模式

- 只看成茶外形，不追溯采制过程。
- 为方便演示省掉关键干燥和保存逻辑。

### 盲点 / 时代局限

- 《茶经》主要对应唐代团饼茶体系。
- 现代绿茶、乌龙、红茶等工艺不能直接用本流程覆盖。
