    ## 1. 结构

    ### 类型

    用户体验架构方法论

    ### 一句话主旨

    用战略、范围、结构、框架、表现五个层面，把用户体验从抽象感受拆成可诊断的产品设计系统。

    ### 骨架

    1. 战略层：产品目标与用户需求。
2. 范围层：功能规格与内容需求。
3. 结构层：交互设计与信息架构。
4. 框架层：界面、导航与信息设计。
5. 表现层：感知设计。

    **论点之间的关系**: 层层递进，并以产品问题为中心转为可调用方法。

    ### 作者要解决的核心问题

    如何把书中的经验、框架和案例转化为真实产品判断中的行动方法，而不是停留在故事或概念层。

    ## 2. 解释

    ### 关键术语

    | 术语 | 作者用法 | 和常识用法的差异 |
    |---|---|---|
    | 五层模型 | 战略、范围、结构、框架、表现。 | 不是线性瀑布，而是相互约束。 |
| 战略层 | 用户需求和产品目标。 | 不是商业口号。 |
| 范围层 | 功能与内容范围。 | 不是需求池。 |
| 结构层 | 信息和交互组织方式。 | 不是视觉布局。 |
| 表现层 | 用户最终感知到的界面。 | 不是美化。 |

    ### 核心命题

    1. 用户体验问题可以按层拆解。
2. 下层不清会污染上层。
3. 功能规格要具体可判断。
4. 界面设计、导航设计和信息设计不同。
5. 表现层应服务于前四层。

    ### 论证链

    作者通过章节框架、案例和反例，把产品判断从抽象观点推进到可操作的取舍：先解释问题结构，再展示案例，最后给出行动边界。

    ## 3. 批判

    ### 作者的时代局限 / 立场盲点 / 未证明假设

    - 模型源自 Web 与软件体验，现代多端、AI 和服务体验需扩展。
- 五层容易被误解成瀑布流程。
- 强调结构化可能低估情感和品牌因素。

    ### 最强反对意见

    这些方法来自特定行业、时代和作者经验，不能替代目标行业的事实调研、数据实验、合规审查和团队执行能力。
