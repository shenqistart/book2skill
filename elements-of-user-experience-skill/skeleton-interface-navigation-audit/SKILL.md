# 框架层界面导航审计

## R — 来源依据 (Reading)

> 原书依据见本节来源说明。

依据界面设计、导航设计和信息设计章节。

本仓库不收录原书全文；此处只保留章节级依据和方法论重述。

## I — 方法论骨架 (Interpretation)

- 框架层把结构落到屏幕。
- 界面设计关注控件，导航设计关注移动，信息设计关注理解。
- 好框架让用户知道能做什么、在哪里、下一步去哪。

## A1 — 书中的应用 (Past Application)

### 案例 1: 导航与信息设计
- **问题**: 同一页面同时有控件、路径和信息说明。
- **方法论的使用**: 分开审计三类设计。
- **结论**: 降低混乱。
- **结果**: 页面更可用。

## A2 — 触发场景 (Future Trigger)

### 用户会在什么情境下需要这个 skill?

当页面元素、导航、按钮和信息呈现已经成形，需要检查是否支撑用户任务时调用。不适用于重新定义产品战略。

### 语言信号

- "这个页面布局合理吗？"
- "导航怎么设计？"
- "用户不知道下一步"

### 与相邻 skill 的区分

- 本 skill 处理 `框架层界面导航审计` 这一单点问题；若问题跨越多个环节，请先读本书 `INDEX.md` 选择组合顺序。

## E — 可执行步骤 (Execution)

1. **审计控件**
   - 完成标准: 检查主要动作是否清楚可用。

2. **审计导航**
   - 完成标准: 检查当前位置、去向和返回。

3. **审计信息**
   - 完成标准: 检查用户是否理解内容和状态。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 不要只凭审美改框架层。
- 移动端和桌面端约束不同。

### 作者的盲点 / 时代局限

- 模型源自 Web 与软件体验，现代多端、AI 和服务体验需扩展。
- 五层容易被误解成瀑布流程。
