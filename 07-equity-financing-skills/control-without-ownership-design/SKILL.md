# 分股不分权设计

## R — 原文 (Reading)

> “分‘钱’而不分‘权’”
>
> 来源: 李利威《一本书看透股权架构》第2章。

## I — 方法论骨架 (Interpretation)

这个 skill 把“给多少股”拆成两个问题: 给经济利益, 还是给治理权。书中把控制权设计整理为七类工具: 有限合伙企业、金字塔架构、一致行动人、委托投票权、公司章程控制、优先股和 AB 股。

选择工具时不要先追求复杂结构, 而要先看四件事: 被激励对象是否需要表决权, 是否需要长期绑定, 公司是否能承受税务和管理成本, 未来融资或上市路径是否接受这种安排。

它的目标不是把控制权永远锁死, 而是在激励、融资、治理和退出之间找到可解释的结构。

## A1 — 书中的应用 (Past Application)

### 案例 1: 员工持股平台

- **问题**: 公司希望让员工分享收益, 又不希望股东会人数过多或表决权分散。
- **方法论的使用**: 通过有限合伙平台集中持有员工权益, 由普通合伙人控制平台表决。
- **结论**: 员工获得经济利益, 但表决权集中在平台控制人。
- **结果**: 适合员工激励, 但需要额外关注税务、退出和平台治理。

### 案例 2: 海康威视式持股平台讨论

- **问题**: 员工大规模参与持股后, 如何兼顾激励和控制。
- **方法论的使用**: 用平台承接员工持股, 减少直接股东数量并统一表决。
- **结论**: 持股平台可以降低治理复杂度, 但不是零成本工具。
- **结果**: agent 需要同时提醒税务、合伙协议、退出条件和信息披露。

### 案例 3: 一致行动与委托投票

- **问题**: 创始团队股权分散, 但希望对外保持统一决策。
- **方法论的使用**: 在股权比例不变的情况下, 通过协议统一投票方向。
- **结论**: 钱权可以阶段性分离, 但协议期限、违约责任和撤销机制必须清楚。
- **结果**: 避免只靠口头信任维持控制。

## A2 — 触发场景 (Future Trigger)

### 用户会在什么情境下需要这个 skill?

1. 用户想做员工持股、期权、合伙人激励或资源方分成。
2. 用户担心“给了股就给了表决权”。
3. 用户正在比较有限合伙、代持、委托投票、一致行动或 AB 股。
4. 用户需要向投资人解释为什么股权结构不会造成治理失控。

### 语言信号

- “怎么分股不分权?”
- “员工持股平台怎么设计?”
- “给资源方股权会不会影响控制?”
- “一致行动人和委托投票有什么区别?”

### 与相邻 skill 的区分

- 与 `founder-control-threshold-map` 的区别: 本 skill 用于设计工具; 前者先判断现有权力边界。
- 与 `valuation-dilution-clarifier` 的区别: 本 skill 不计算估值和稀释, 只处理钱权结构。

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **拆分利益和权力**
   - 完成标准: 明确对象需要的是分红、增值、退出收益、参与治理、信息权还是任免权。

2. **限定约束条件**
   - 完成标准: 记录公司阶段、主体类型、未来融资/上市设想、税务敏感度、员工人数和退出安排。

3. **筛选工具**
   - 完成标准: 至少比较 3 类工具, 并说明为什么排除其他工具。

4. **设计权利表**
   - 完成标准: 输出经济权、表决权、信息权、退出权、违约责任和争议解决六列。

5. **做风险复核**
   - 完成标准: 标出税务、合规、代持争议、协议可执行性、投资人接受度和小股东保护风险。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 用户只是问融资估值、投资金额或期权池稀释。
- 用户要求规避税收、隐匿股东、绕开监管或欺骗投资人。
- 用户的问题已经进入具体诉讼或仲裁。

### 作者在书中警告的失败模式

- 把员工激励简单理解成直接分股。
- 忽视持股平台的税务和退出成本。
- 只设计控制权, 不设计退出和违约。

### 作者的盲点 / 时代局限

- AB 股、优先股和上市地规则变化较快, 必须用最新规则复核。
- 有限合伙和代持结构的税务处理依赖所在地和具体交易。

### 容易混淆的邻近方法论

- “控制权阈值”: 本 skill 设计工具前应先调用 `founder-control-threshold-map`。
- “期权激励方案”: 本 skill 只处理钱权分离逻辑, 不替代完整薪酬方案。
