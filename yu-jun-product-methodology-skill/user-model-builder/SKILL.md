# 用户模型构建

## R — 来源依据 (Reading)

> 原书依据见本节来源说明。

依据“用户模型——产品经理要研究用户行为及其背后的原理”“用户不是自然人，而是需求的集合”等章节。

本仓库不收录原书全文；此处只保留章节级依据和方法论重述。

## I — 方法论骨架 (Interpretation)

- 用户不是一个抽象自然人，而是在具体场景下带着多个需求的人。
- 产品经理要解释用户行为背后的目标、约束、心理和环境。
- 用户模型不是贴标签，而是能预测用户下一步会怎么选。
- 好的用户模型要把现实世界、宏观背景和微观场景连接起来。

## A1 — 书中的应用 (Past Application)

### 案例 1: 打车场景用户行为
- **问题**: 乘客醉酒、遗失物品、修改目的地等场景行为复杂。
- **方法论的使用**: 把用户行为放回具体场景和约束中分析。
- **结论**: 不同用户状态下应设计不同规则。
- **结果**: 书中用多个出行案例展示用户模型。

### 案例 2: 搜索和体验文章
- **问题**: 早期搜索用户体验不能只看结果页功能。
- **方法论的使用**: 观察用户完整过程和不同时期认知变化。
- **结论**: 用户体验因人、因时而变。
- **结果**: 附录用搜索案例说明用户模型。

## E — 可执行步骤 (Execution)

1. **定义需求集合**
   - 完成标准: 列出同一用户在场景中的多个需求，而不是一个标签。

2. **还原行为机制**
   - 完成标准: 解释触发、动机、约束、替代方案。

3. **建立预测**
   - 完成标准: 说明这个模型能预测什么选择。

4. **找验证数据**
   - 完成标准: 列出需要观察、访谈或实验验证的证据。

## B — 边界 (Boundary)

### 不要在以下情况使用此 skill

- 不要把用户模型当静态画像。
- 不能用单个高声量用户代表整体。
- 隐私和敏感人群研究需合规。

### 作者的盲点 / 时代局限

- 书中大量互联网平台和出行案例，对硬件、医疗、教育等行业需要重新校准。
- 经济学语言很强，可能低估审美、文化和身份认同类产品的非理性价值。
